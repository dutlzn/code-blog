jvm-年轻代、老年代垃圾回收算法

java jvm
jvm
发布日期:   2021-11-18

年轻代垃圾回收算法

年轻代的内存分为两块

如果根据GC roots 标记哪些对象可以被垃圾回收 就直接对那块内存区域中的对象进行垃圾回收，容易造成内区东一块 西一块，大量的内存碎片。

内存碎片会造成内存浪费。容易导致没有足够的空间来分配新的对象。很多内存碎片没法使用。

可以把新生代的内存区域分为两块，把存活的对象转移到另一块空白的内存中。然后把需要清理的垃圾一块清理掉，这样可以避免内存碎片的产生。

按照这种方式，两块内存区域这么重复循环使用。

缺点：只有一半的内存空间可以使用，对内存使用效率太低了。

复制算法的优化 Eden Survivor

一次垃圾回收过后，大部分对象都要回收。

这里存在一个假设，每次只有少量对象可以存活下来。

可以把年轻代分为80%的Eden， 两个20%的Survivor区域

刚开始对象都是分配在Eden，如果Eden满了，就会触发垃圾回收。

Eden存活对象一次性转移到Survivor，Survivor存放的是上一次Young GC后婚活的对象。

如果Eden又满了，Eden再一次Young GC存活的对象和上一次存活的对象转移到另一块Survivor中。

好处：只有10%的内存空间是被闲置的。90%的内存空间都是被利用的。

当新生代的对象躲过了新生代垃圾回收15次（这个参数可以自己设定）

未来能更好适应不同程序的内存状况，虚拟机并不总是要求对象的年龄必须到达阈值，才能晋升老年代。

动态对象年龄判定：如果Survivor区域中相同年龄所有对象大小的综合大于Survivor内存大小的一半，年龄大于等于该年龄的对象就可以直接进入老年代。

大对象直接进入老年代：-XX:PretenureSizeThreshold 避免大对象在内存里来回复制，耗费时间

经历过多次Young GC之后，对象变得很多，无法放入另一块Survivor，这个时候需要把这些对象直接转移到老年代。

老年代空间分配担保原则

如果年轻代大量对象存活下来，Survivor区域放不去，老年代区域也放不下该咋办？

执行YoungGC之前，JVM会检查老年代的可用内存空间大小是否大于年轻代所有对象的总大小。

  • 如果老年代可用内存空间大小大于新生代所有对象，可以放心Young GC 。大不了全部放在老年代里。
  • 如果发现老年代空间不够：
    • 查看-XX:- HandlePromotionFailure 参数是否设置
    • 如果参数已经被设置了，若老年代的内存大小大于之前每一次Young GC后进入老年代的对象的平均大小，那么老年代空间是够的。
    • 参数没有设置或者参数设置发现还是不够，就会触发Full GC

Young GC满足条件

两个步骤：

  • 老年代内存>新生代
  • 老年代内存>Young GC转移平均大小

这两个条件都满足了。可以尝试Young GC

Young GC的可能结果

  1. YoungGC 过后，年轻代所有存活对象是小于Survivor区域大小。那么存活对象进入Survivor区域
  2. YoungGC之后，年轻代所有存活对象是大于Survivor区域大小的，但是小于老年代区域。就进入老年代
  3. YoungGC之后，年轻代>survivor 而且老年代也不够。触发FullGC

Full GC 不仅对老年代进行垃圾回收，也会对年轻代进行垃圾回收。Full GC失败就会报OOM异常。

老年代垃圾回收算法

老年代采用的是标记整理算法
老年代会标记哪些对象可以存活，然后进行移动，防止出现过多的内存碎片，最后一次性把垃圾对象都回收掉。老年代的垃圾回收算法很耗时。

如果系统频繁进行老年代的垃圾回收，会严重影响系统性能，出现系统卡顿的情况。
