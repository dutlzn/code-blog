jvm-垃圾回收器之G1 GC

java jvm
jvm
发布日期:   2021-11-19

G1 垃圾回收器的基本原理

ParNew + CMS组合有哪些不足

最大的问题在于会暂停系统工作线程，会影响系统的性能

G1 垃圾回收器

  • G1可以同时回收老年代和新生代

  • G1将java堆内存分成多个大小相等的Region

  • 然后G1也会有新生代和老年代的概念，但是只不过是逻辑上的概念

  • G1最大的特点就是，可以设置一个垃圾回收的预期停顿时间，可以直接控制垃圾回收对系统性能的影响

G1如何做到对垃圾回收导致的系统停顿可控

针对每一个Region对其垃圾回收时间进行判断。必须要追踪每个Region里的回收价值。它必须搞清楚每个Region里的对象有多少是垃圾，如果对这个Region进行垃圾回收，需要耗费多长时间，可以回收掉多少垃圾？根据时间要求自己去决定去处理哪一个垃圾。

Region可能属于新生代也可能属于老年代

  • Region随时会属于新生代也会属于老年代，所以没有所谓新生代给多少内存，老年代给多少内存这一说了
  • 实际上新生代和老年代各自的内存区域是不停的变动的，由G1自动控制。
  • Region动态转移给新生代或者老年代，按需分配
  • 触发垃圾回收的时候，可以根据设定的预期系统停顿时间，来选择最少回收时间和最多回收对象的Region进行垃圾回收，保证GC对系统停顿的影响在可控范围内，同时还能尽可能回收最多的对象

如何设定G1对应的内存大小

G1 对应的一大堆的Region内存区域，每个Region大小是一致的

两个问题

  • 有多少个Region
  • 每个Region大小是多少

JVM启动的时候一旦发现使用的是G1垃圾回收器，就会自动用堆大小2048。因为JVM最多可以有2048个Region Region的大小必须是2的倍数

刚开始的时候，默认新生代堆内存的占比是5%，100个region 这个占比是可以通过设置参数来调整的

在系统运行中，jvm会不停给新生代增加更多的Region，但是最多新生代的占比不会超过60%。这个最大也是可以通过参数来设置的。一旦Region进行了垃圾回收，此时新生代的Region数量还会减少，这些都是动态的

新生代还有Eden和Survivor的概念吗

G1把内存划分为了很多的Region， 但是还是有很多新生代、老年代，新生代里也有Eden Survivor划分的 之前的一些原理也适用于G1

比如新生代有100个Region 其中80个是Eden 两个Survivor 各自占10个。这个比例也是通过手动设置参数来设定的

随着对象不停的在新生代里分配，属于新生代的Region 会不断增加变化。Eden Survivor所对应的Region 也会不断增加

G1新生代垃圾回收

G1 新生代也有Eden Survivor 的区分， 所以触发垃圾回收的机制类似的。一旦新生代达到了设定的占据堆内存最大大小60%，这个时候会触发新生代的GC，G1就会用复制算法来进行垃圾回收。然后把Eden对应的Region中的存活对象放入S1对应的Region中，接着回收掉Eden对应的Region中的垃圾对象。

这个过程中还是和之前的新生代垃圾回收有一些区别：

G1可以设定目标GC停顿时间的，也就是G1执行GC的时候最多可以让系统停顿多长时间
可以通过参数来设定

G1会对每个Region进行追踪回收，可以回收多少对象来选择回收一部分的Region，保证GC停顿时间控制在指定范围内，尽可能多回收掉一些对象

对象什么时候进入老年代

对象在新生代躲过了很多次的垃圾回收，达到了一定的年纪了，就会进入老年代，可以通过参数来设置
动态年龄判定规则，如果一旦发现某次新生代GC之后，存活对象超过了Survivor的50%
此时就会判断一下，比如年龄为1岁，2岁，3岁，4岁的对象的大小总和超过了Survivor的50%，此时4岁以上的对象全部会进入老年代，这就是动态年龄判定规则

大对象Region

G1提供了专门的Region存放大对象，而不是让大对象进入老年代的Region中
大对象的判定规则是 大对象超过了一个Region的50%。如果大对象实在太大，可以横跨多个Region来存储
新生代、老年代是不断变化的，这个变化过程中有一些region腾出来了就可以存放大对象。
新生代、老年代在进行垃圾回收的时候，会带着大对象Region一起回收。

什么时候触发新生代+老年代混合垃圾回收

有一个参数可以设置 比如设置45% -XX:InitiatingHeapOccupancyPercent。如果老年代占据了堆内存的45%的Region的时候，此时就会尝试触发一次新生代+老年代的混合回收阶段

总结

G1动态内存管理策略 会根据情况动态把Region分配给新生代、Eden、Survivor、老年代和大对象Region
但是新生代和老年代有一个各自的最大占比，然后在新生代的Eden满的时候，会触发新生代的GC
新生代的垃圾回收还是采用复制算法，只不过会考虑停顿时间，保证垃圾回收的停顿时间不会超过预设值，通过手动输入的参数来决定每次GC停顿时间。这样就会导致每次GC只会挑选一部分Region去进行垃圾回收，追踪回收，多少大小、耗时多少综合决定是否回收这个Region
新生代对象如何进入老年代： 动态年龄判定 新生代对象躲过了15次新生代的垃圾回收，就进入老年代， 后者 新生代存活对象在Survivor放不下了
大对象直接进入单独的Region 不再进入老年代

G1垃圾回收过程

  • 初始标记，这个阶段，暂停系统工作线程，标记GC roots直接引用的对象。这个过程速度很快先停止系统程序的运行，然后对各个线程栈内存中的局部变量的GC roots，以及方法区中的类静态变量代表的GC roots进行扫描，标记出来它们直接引用哪些对象
  • 并发标记：这个阶段允许系统程序的运行，同时进行GC roots追踪,从GC roots开始追踪所有存活对象。这个过程很耗时，因为要追踪所有存活对象。但是这个阶段是可以跟系统工作线程并发运行的，所以对系统程序的影响不大。
  • 最终标记阶段：这个阶段暂停系统工作线程，根据并发标记记录的哪些对象修改，最终标记一下有哪些存活对象，有哪些是垃圾对象。
  • 混合回收：这个阶段会计算老年代中每个Region中的存活对象数量，存活对象的占比，还有执行垃圾回收的预期性能和效率。接着会停止系统程序，然后全力以赴尽快进行垃圾回收，此时会选择部分Region进行回收，因为必须让垃圾回收的停顿时间控制在我们指定的范围内。老年代对堆内存占比达到45%的时候，触发的是“混合回收”也就是说，此时垃圾回收不仅仅是回收老年代，还会回收新生代，还会回收大对象。
